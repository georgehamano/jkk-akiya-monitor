# JKK東京 空き家（募集）監視システム 要件定義書

| 項目 | 内容 |
|------|------|
| 文書名 | 要件定義書 |
| 対象 | JKKねっと（東京都住宅供給公社）空き家・募集状況の変化検知とLINE通知 |
| 想定利用 | Google Colab 定期実行、将来的に GitHub Actions |

---

## 1. 背景・目的

### 1.1 背景
JKK東京のインターネット申込サイト（JKKねっと）上で公開される**空き家・募集戸数**の情報が変化したとき、手動で確認するのは負荷が高い。

### 1.2 目的
- 対象ページから**物件単位の在庫（募集戸数の合算）**を定期的に取得する。
- **前回実行時と比較**し、**在庫が増えた場合**および**新規に掲載が出た場合**に限定して**LINEで通知**する。
- 同一物件で**合計件数は変わらないが、間取り別の内訳が入れ替わった場合**（例: 30→30）も検知し通知対象とする。
- サイトが**メンテナンス・おわび画面**のときは、例外で落とさず**ログ出力して正常終了**する。

---

## 2. ステークホルダー・利用シナリオ

| 役割 | 期待 |
|------|------|
| 運用者 | 定期実行（Colab / GHA）で通知のみ受け取りたい |
| 開発者 | `requests` のみで実装し、環境変数で秘密情報・URLを切り替え可能にしたい |

---

## 3. スコープ

### 3.1 対象とする範囲（In Scope）
- HTTP(S) で HTML を取得し、一覧表から**住宅名・間取り（行識別子）・募集戸数**を抽出する。
- **物件名（住宅名）→ 当該物件の募集戸数合算**を `last_data.json` に保存する。
- **前回保存データ**と比較し、通知条件を満たす場合のみ **LINE Messaging API（Push）** で通知する。
- **おわび・メンテナンス**相当画面の検知と、**一覧が取れない**場合の扱い（仕様書で詳細化）。

### 3.2 対象外（Out of Scope）
- ブラウザ自動操作（Playwright / Selenium）を**必須**としない（要件上は `requests` のみ。取得不能時は「別手段の検討」を将来課題とする）。
- JKK側の利用規約違反となる過度な高頻度アクセス（レートは運用で調整）。
- 申込処理・ログイン後のマイページ操作の自動化。

---

## 4. 機能要件

### 4.1 データ取得
| ID | 要件 |
|----|------|
| FR-01 | `requests.Session()` を用い、**先に公社公式サイト等へアクセスして Cookie を取得したうえで**、監視対象 URL を取得すること。 |
| FR-02 | User-Agent, Referer, Accept-Language 等を**一般的なブラウザに近い**値で付与すること。 |
| FR-03 | 文字コードは **CP932 / Shift_JIS / UTF-8** 等を考慮し、一覧の日本語列名が正しく読めるデコード結果を優先すること。 |
| FR-04 | 監視対象 URL は**環境変数で変更可能**とすること（既定値は別紙仕様）。 |
| FR-05 | `https://jhomes.to-kousya.or.jp/` のルートは **404** となるため、ウォームアップ URL として使わないこと。 |

### 4.2 HTML 解析・正規化
| ID | 要件 |
|----|------|
| FR-10 | 一覧は **`table.cell666666`** を主対象とし、ヘッダに **住宅名・間取り・募集戸数**（または同等の列）を認識すること。 |
| FR-11 | ヘッダ行が先頭行でない場合、**先頭数行をスキャン**してヘッダ行を特定できること。 |
| FR-12 | ヘッダ文字列が HTML 実体参照のみの場合でも、`html.unescape` 等で認識できること。 |
| FR-13 | ヘッダ認識に失敗する場合、**固定列インデックス**（外観=0, 住宅名=1, …, 募集戸数=9）によるフォールバックを可能とすること。 |
| FR-14 | 同一**住宅名**が複数行（間取り別）存在する場合、**募集戸数は合算**して物件単位の件数とすること。 |
| FR-15 | 詳細ボタンは `senPage('p1','p2','p3','p4')` 形式の `onclick` を解析し、**詳細 URL テンプレート**（環境変数）に埋め込めること。 |

### 4.3 中間ページ・遷移
| ID | 要件 |
|----|------|
| FR-20 | 「数秒後に自動で次の画面」等の**中間ページ**を検知した場合、**遷移先 URL の抽出**（meta refresh / script / リンク / 同一 HTML 内 URL）を試みること。 |
| FR-21 | **同一 URL の無限ループ**を避けるため、訪問済み URL を管理すること。 |
| FR-22 | URL 候補が得られない、または不十分な場合、**`<form>` の GET/POST 送信**によるフォールバックを試みること（実装可能性はサイト構造に依存）。 |

### 4.4 差分・通知条件
| ID | 要件 |
|----|------|
| FR-30 | `last_data.json` に **物件名（住宅名）→ 合算募集戸数** の辞書を保存すること。 |
| FR-31 | **入れ替わり検知**用に、物件ごとの行シグネチャ（間取り・募集戸数・senPage 等）のハッシュを別ファイル（例: `last_rooms.json`）に保存すること。 |
| FR-32 | 次のいずれかを満たす物件を**通知対象**とする: (a) 前回より合算件数が増加、(b) 前回存在しなかった物件の出現、(c) 合算件数は同じだが行シグネチャが変化（内訳入れ替わり）。 |
| FR-33 | LINE メッセージに **物件名・増加件数（入れ替わりは 0 可）・現在の合算件数・詳細 URL** を含めること。 |
| FR-34 | LINE のチャネルアクセストークン・送信先 ID は**環境変数**で与え、未設定時は通知をスキップしログに明示すること。 |

### 4.5 メンテナンス・異常時
| ID | 要件 |
|----|------|
| FR-40 | おわび画面は **文字列・タイトル・画像ファイル名**（例: `owabimoji8.gif`）等で検知し、**「メンテナンス中」ログで終了**すること（プロセスは異常終了としない）。 |
| FR-41 | 一覧行が 0 件の場合、**デバッグ用 HTML 保存**とヒントログを出力し、異常終了としないこと。 |

### 4.6 保存場所
| ID | 要件 |
|----|------|
| FR-50 | 状態ファイルの保存先は環境変数 `JKK_DATA_DIR` で指定可能とすること。 |
| FR-51 | 未指定時は**スクリプトファイルと同一ディレクトリ**に保存すること。 |

---

## 5. 非機能要件

| ID | 要件 |
|----|------|
| NFR-01 | HTTP クライアントは **`requests` のみ**とする（BeautifulSoup は HTML パース用途で可）。 |
| NFR-02 | タイムアウトを設定し、長時間ブロックしないこと。 |
| NFR-03 | 秘密情報をリポジトリにコミットしないこと（環境変数・Colab シークレット・GHA Secrets）。 |
| NFR-04 | Colab と GitHub Actions の両方で実行可能な構成（パスは環境変数で吸収）とすること。 |

---

## 6. 制約・前提

- 監視対象サイトの HTML 構造・URL は**予告なく変更**されうる。変更時はパーサー・監視 URL の見直しが必要。
- データセンター出口 IP 等により、**一覧が返らない・おわびのみ**となる可能性がある（JKK側ポリシー・負荷による）。
- 法的・利用規約上の遵守は運用者の責任で確認すること。

---

## 7. 用語

| 用語 | 説明 |
|------|------|
| 住宅名 | 一覧表の物件（団地・住宅）名称 |
| 募集戸数 | 当該行（間取りタイプ等）の募集戸数 |
| 合算件数 | 同一住宅名の全行の募集戸数の合計 |
| おわび画面 | メンテナンス等で一覧の代わりに表示される案内ページ |
